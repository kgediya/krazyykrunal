SnapML adds perception to your Lens

What SnapML does

SnapML lets you run machine learning models directly inside the Lens, so your camera feed becomes something your experience can understand and react to.

Why object detection

You are not just classifying a frame. You are locating objects with bounding boxes and reacting to them spatially on screen.

Creative payoff

Once you know where the food is, you can attach UI, stickers, hints, and game logic exactly where it appears.

camera texture ML component raw outputs processed boxes interactive visuals

What SnapML really means

You may already be using machine learning in Lens Studio through segmentation, skeletal tracking, and other built-in features. SnapML extends that by letting you bring in your own models, so you can go beyond the default capabilities of Lens Studio.

Those Lenses still ship like normal Snapchat Lenses, which means people do not need to install anything extra to use them.

How to think about a model

In Lens Studio, a model behaves like a black box. You tell Lens Studio what goes in, such as camera texture or cropped input, and then you tell it how to use what comes out, such as probabilities, coordinates, or images.

  • Inputs can be textures, processed image crops, or other model-ready data.
  • Outputs can be numbers, labels, box coordinates, masks, or full textures.
  • That means SnapML supports both analysis and visual effects workflows.

Supported formats

Lens Studio supports .ONNX and .TFLite models, and you can bring in models created with tools such as PyTorch, TensorFlow, and Roboflow as long as they are exported in a compatible format.

ML Component

The ML Component is where you define the model inputs, outputs, and run behavior: every frame, on demand, or in another thread so your Lens can stay responsive.

Performance story

SnapML handles optimized execution for you and can use device-specific hardware acceleration when it is available.

Import the Food & Fruits detection model

What it detects

The bundled model can detect foods and fruits such as banana, apple, sandwich, orange, broccoli, carrot, hot dog, pizza, donut, and cake.

banana apple orange broccoli pizza donut cake carrot

Import path

  1. Open the Object Detection example or template.
  2. Find the Food Detection model in the example assets.
  3. If you are using your own model, drag the `.onnx` file into Asset Browser.
  4. Select the object that contains the ML Component.
  5. Assign the food model in the Model field.

Replace the existing model without breaking the pipeline

Input setup

Use the device camera texture and keep the detector-friendly transformer setup shown in the docs.

  • Input shape: 128 × 256 × 3
  • Stretch: off
  • Horizontal/Vertical align: center
  • Rotation: none
  • Fill color: black

Output setup

If your model follows the example structure, keep the output names expected by `MLController`. If not, map the output location and class score names correctly.

Preview prep

Load a preview image or video that clearly shows fruit or food, so you can verify detections before you style the Lens.

Build this: Food Mouth Lens

As soon as a food item is detected, your Lens attaches an animated mouth sticker to it so the snack feels alive on screen. The result is simple, funny, and realistic to build inside the Object Detection template.

  • An apple can grin, chew, or lip-sync with a looping mouth sticker.
  • A donut can get a tiny floating mouth in the center hole for comic effect.
  • A pizza slice can look like it is talking back to the user.
  • If multiple foods appear, each one can get its own mouth for a chaotic scene.
  • You reuse detection box anchors to place a mouth sticker inside each detected box.
  • You replace the default visual children with an animated mouth image or sprite.
  • You keep the mouth centered or slightly lower within the detection area for better placement.
  • You can scale or switch mouth variants based on the detected class label.
  • No retraining required if using bundled food model.
  • Most of the work is 2D UI, sticker animation, and anchor positioning.
  • It is much lower risk than trying to attach full 3D content to food items.
  • You can prototype it quickly by editing the detection box children already in the template.
  • It works well for a workshop because the result is playful but technically realistic.

Train and bring in your own model

What is supported

If you use the Roboflow Lens Studio deployment flow, the supported task here is object detection. Roboflow also notes that only models trained with Roboflow Train 3.0 are supported for this export path.

What you do in Roboflow

  1. Create or import your dataset.
  2. Annotate and generate a version.
  3. Train your detector.
  4. Open Deploy and choose Export to Lens Studio.

What gets downloaded

You get the Roboflow Lens Studio template plus your model weights. Keep the class list shown during export because you will need that exact order inside Lens Studio.

What you configure in Lens Studio

  • Open the Roboflow Lens Studio template or your own object detection project.
  • Select the ML Controller.
  • Click ML Model and import the exported weights.
  • In the import popup, set each input Scale to 0.0039.
  • Leave bias values at their defaults unless your export says otherwise.

One detail you cannot skip

In Class Settings, enter every class name and label in the exact same order shown in Roboflow. If the order is wrong, your detections will map to the wrong labels.

Fast mental model: weights + correct scale + correct class order = usable custom detector.

Use vibe coding to move faster

What vibe coding looks like here

You keep one Codex workspace that contains your custom training notebooks, SnapML docs, Lens Studio references, exported models, and experiment notes. Instead of context switching across ten tabs, you build with everything close by.

With Codex Environment, you can keep notebooks, docs, exported models, and implementation work in one workspace.

Why it helps

  • You can move from idea to prototype without losing context.
  • You can compare docs, notebooks, and model outputs side by side.
  • You can reuse prompts, snippets, class lists, and config notes.
  • You can iterate on custom detectors much faster.

A practical vibe-coding loop for SnapML: keep research, training, and Lens iteration in one working environment.

Three simple techniques

  • Keep a starter prompt library for repeat jobs like model swapping and class mapping.
  • Save every training run with notes on preprocessing, export settings, and preview results.
  • Treat your workspace like a lab bench: docs, notebooks, Lens output, and your assistant in one loop.

How to keep it useful

A strong vibe-coding setup does not replace understanding. It compresses the boring parts: setup, recall, cross-referencing, and repetition. You still make the product decisions.

  • Ask for quick script scaffolds, then inspect and adapt them.
  • Keep your class order, thresholds, and import settings documented.
  • Use the AI to summarize docs, not to replace testing.
  • Always validate your model in Preview and on-device pairing.
